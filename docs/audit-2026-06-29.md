# DraftHQ end-to-end audit

Date: 2026-06-29

## Executive summary

DraftHQ already has the right core architecture for a reliable fantasy draft product: draft order, expected-pick validation, pick uniqueness, lifecycle changes, and clock deadlines are enforced in Postgres RPCs rather than trusted to React. The league, season, team, lobby, live room, commissioner controls, invitations, branding, imports, and persistent/anonymous identity flows are all represented in the product.

It is not production-ready yet. The release gate is red, commissioner authority is represented inconsistently, and several storage/API surfaces are more permissive than the product rules allow. The most urgent work is authorization consolidation and test-environment repair—not additional music, presentation, or theme work.

Verdict: **promising core, pre-production security/reliability hardening required**.

No application code was changed during this audit.

## Existing product and technical structure

- Next.js 16 App Router with public auth pages, dashboard, league workspace routes, team/season/settings pages, a join flow, a pre-draft lobby, and the live draft room.
- Supabase Auth supports anonymous draft participants and persistent commissioner/member accounts. Signup upgrades an anonymous identity, preserving its user ID.
- A league identity layer (`leagues`, `league_members`, `league_teams`, seasons) sits above the older draft layer (`drafts`, `teams`, `draft_participants`, `picks`). This is the source of most authorization drift.
- Core draft mutations use `security definer` Postgres functions, row locks, expected-pick arguments, uniqueness constraints, and server-owned deadlines. React primarily renders snapshots and invokes RPCs.
- The live room subscribes to draft/team/participant/pick changes, tracks presence, reconciles by re-fetching a full room snapshot, polls the draft revision, and periodically re-syncs server time.
- Snake behavior and timer calculations are factored into testable libraries. The draft room and team setup surfaces themselves are very large client components.
- Sleeper draft import and history sync are implemented. ESPN, Fleaflicker, MFL, and Yahoo season creation previews/imports are implemented. ESPN/Yahoo league-history integration remains presented as coming soon, which is distinct from season creation.
- The pre-draft lobby exists as `/draft/lobby` and reuses `DraftRoom` with `lobbyOnly` rather than maintaining a separate state model.
- League/team logos, owner photos, colors, banners, walk-up music, sound effects, and presentation settings are present. Classic remains the only exposed draft-room theme.

## What is working well

- Snake positions and round transitions are deterministic and unit-tested.
- Picks are protected by database locks, expected-pick checks, draft/player uniqueness, and lifecycle checks.
- The clock uses a server deadline and a measured server-time offset; it is not authoritative in `localStorage`.
- Service-role access is isolated behind `server-only`, and no tracked environment file contains deployed secrets.
- Join previews disclose limited data through a dedicated RPC, while private draft data remains behind membership checks.
- Most commissioner mutations already check `drafts.commissioner_user_id` or the league commissioner helper in the database.
- Build output successfully compiled all 35 application/API routes.

## Critical bugs and security issues

### C1. Stale draft participant roles can retain destructive commissioner power

- **File/path:** `supabase/migrations/20260624000000_m5_fixes.sql`, `supabase/migrations/20260629000007_transfer_league_ownership_v2.sql`, `supabase/migrations/20260620010000_add_commissioner_recovery.sql`
- **What is wrong:** Draft authority has two competing sources: `drafts.commissioner_user_id` and `draft_participants.role = 'commissioner'`. The current `reset_draft` function and pick DELETE policy accept either source. A read-only check of current data found an actual draft where these identities differ after ownership changes.
- **Why it matters:** A former/stale commissioner participant can reset a draft or delete picks after authority has transferred. This is a draft-integrity and authorization vulnerability.
- **Recommended fix:** Make `drafts.commissioner_user_id` the sole draft commissioner authority. Replace participant-role fallbacks in destructive functions/policies, repair existing rows, and add an invariant/transfer routine that keeps the participant projection synchronized without granting authority from it.
- **Can safely fix now:** **No.** The SQL change is straightforward, but it needs a data-repair migration and authorization regression tests before deployment.

### C2. Draft asset and audio storage policies allow unrelated users to overwrite content

- **File/path:** `supabase/migrations/20260625100000_draft_team_assets.sql`, `supabase/migrations/20260628000006_draft_audio_bucket.sql`
- **What is wrong:** Any user with the `authenticated` database role can insert or update any draft team logo/owner photo object. Any authenticated user can insert, update, or delete any draft audio object. Supabase anonymous users also commonly operate with the authenticated Postgres role after sign-in. Policies do not bind object paths to a draft, team owner, or commissioner.
- **Why it matters:** A user from another league can overwrite branding or delete commissioner audio if they know or guess an object path. Public buckets make the resulting defacement immediately visible.
- **Recommended fix:** Adopt a strict path convention and storage helper functions that authorize the authoritative draft commissioner or the assigned owner of that specific team. Add both `using` and `with check` clauses, plus explicit delete behavior and cleanup.
- **Can safely fix now:** **No.** Existing object paths and owner-edit requirements must be inventoried before tightening policies.

### C3. Any authenticated user can overwrite global NFL bye-week data

- **File/path:** `supabase/migrations/20260627000000_add_nfl_bye_weeks.sql`
- **What is wrong:** `upsert_bye_weeks` is a `security definer` RPC granted to all authenticated users and checks only that `auth.uid()` exists. It writes global reference data used by every league.
- **Why it matters:** Any signed-in/anonymous session can corrupt bye weeks for all users, affecting rankings and draft decisions.
- **Recommended fix:** Revoke the RPC from `authenticated`; grant it only to `service_role`, or require an explicit platform-admin claim. Validate season bounds, team abbreviations, duplicates, array size, and all values before writing atomically.
- **Can safely fix now:** **Yes, as a small migration**, once the intended administrative caller is confirmed.

### C4. The rankings synchronization endpoint is unauthenticated and writes with service-role authority

- **File/path:** `src/app/api/rankings/sync/route.ts`, `src/app/api/rankings/route.ts`
- **What is wrong:** Anyone can POST to the sync endpoint. It fetches ESPN, derives rankings, and performs a service-role upsert. The public GET endpoint can also trigger the sync path when data is stale. There is no rate limit, shared secret, job signature, or strict year range.
- **Why it matters:** Attackers can cause repeated third-party traffic and privileged database writes, creating quota, cost, and data-quality risk.
- **Recommended fix:** Move synchronization to a scheduled/admin-only job, authenticate internal calls with a signed secret, strictly bound the season, add rate limiting/idempotency, and keep public GET read-only.
- **Can safely fix now:** **No.** The preferred scheduling/deployment mechanism needs a product/infrastructure decision.

### C5. Yahoo OAuth credentials are browser-global rather than bound to the DraftHQ user

- **File/path:** `src/app/api/providers/yahoo/auth/route.ts`, `src/app/api/providers/yahoo/callback/route.ts`, `src/app/api/providers/yahoo/preview/route.ts`
- **What is wrong:** OAuth state contains a user ID, but the callback only compares the raw state cookie and does not revalidate/bind the current Supabase identity. Yahoo tokens are stored in one HTTP-only cookie for the origin. The preview route validates that *some* persistent user is signed in, then uses that cookie.
- **Why it matters:** If two DraftHQ accounts use the same browser profile, the second can query Yahoo using the first account's tokens. This crosses account boundaries and exposes private league data.
- **Recommended fix:** Store encrypted provider tokens server-side keyed to Supabase user ID, verify the user again in the callback, bind state to that identity, support revocation, and never use an origin-global provider token cookie as the authorization source.
- **Can safely fix now:** **No.** This requires a provider-credential table and OAuth flow migration.

## Important product gaps and fragile areas

### I1. Co-commissioner UI permissions do not match database/API permissions

- **File/path:** `supabase/migrations/20260628000020_co_commissioner_role.sql`, `supabase/migrations/20260622000000_add_league_identity_layer.sql`, `supabase/migrations/20260629000008_update_league_settings_with_slug.sql`, `src/lib/leagueApi.ts`, `src/app/leagues/[slug]/WorkspaceLayoutClient.tsx`, `src/app/api/leagues/[leagueId]/members/route.ts`
- **What is wrong:** The client treats `co-commissioner` as manageable, but `is_league_commissioner`, settings RPCs, and member APIs generally accept only owner/`commissioner`.
- **Why it matters:** Co-commissioners see enabled controls that fail with permission errors, and behavior varies by feature.
- **Recommended fix:** Define a single league capability helper and use it in RLS, RPCs, API routes, and UI. Explicitly decide whether co-commissioners can manage members, transfer ownership, delete a league, and manage integrations.
- **Can safely fix now:** **No.** Product capability boundaries must be agreed first.

### I2. Realtime error recovery can leave picks disabled indefinitely

- **File/path:** `src/lib/draftRealtime.ts`, `src/hooks/useRealtimeDraftRoom.ts`
- **What is wrong:** `CHANNEL_ERROR`/`TIMED_OUT` sets error status but does not schedule resubscription with backoff. Recovery occurs only on focus, visibility, or online events. Revision polling may refresh data while connection status stays errored.
- **Why it matters:** A brief websocket failure can leave a visible room current but unable to pick until the user changes focus/reloads.
- **Recommended fix:** Add bounded exponential resubscription, a health state separate from data freshness, and recovery tests that simulate channel failure without browser focus changes.
- **Can safely fix now:** **Yes, bounded**, but it needs realtime hook tests.

### I3. The fallback revision poll does not cover all room state

- **File/path:** `src/hooks/useRealtimeDraftRoom.ts`, `src/lib/draftApi.ts`
- **What is wrong:** The 10-second recovery poll checks only `drafts.updated_at`. Team, participant, and invitation changes do not reliably bump that value.
- **Why it matters:** A missed realtime event can leave ownership, names, presence-related membership, or invitations stale indefinitely while the tab remains active.
- **Recommended fix:** Add a draft-room revision/version bumped transactionally by relevant writes, or poll a cheap composite revision endpoint/RPC.
- **Can safely fix now:** **No.** Best fixed as a small database/API contract rather than more client queries.

### I4. Realtime reconciliation reloads the full player catalog for ordinary changes

- **File/path:** `src/lib/draftApi.ts`, `src/hooks/useRealtimeDraftRoom.ts`, `src/lib/draftRealtime.ts`
- **What is wrong:** Every subscribed change invokes the full snapshot query, including roughly 900 active players, teams, participants, picks, and settings.
- **Why it matters:** Picks, clock changes, team edits, and presence recovery can generate large repeated responses on every connected device. This will increase latency and Supabase egress at real draft concurrency.
- **Recommended fix:** Separate immutable/slow player catalog loading from compact draft-state reconciliation; debounce bursts and update simple events locally before periodic authoritative reconciliation.
- **Can safely fix now:** **No.** This changes the room data contract and deserves performance baselines.

### I5. Pick deletions are broadcast to every connected draft, and invitation changes are not subscribed

- **File/path:** `src/lib/draftRealtime.ts`
- **What is wrong:** DELETE events on `picks` have no draft filter, so every live room refreshes after any draft undo/reset. `draft_invitations` is included in snapshots but has no subscription.
- **Why it matters:** Unrelated drafts cause needless refreshes; commissioner invitation state can be stale across devices.
- **Recommended fix:** Prefer a draft revision/broadcast event that includes draft ID for deletes and invitation changes, or use a secure database trigger/broadcast design.
- **Can safely fix now:** **No.** Supabase DELETE filtering limitations make a database-side event preferable.

### I6. Commissioner recovery migration is pending and lacks a regression test

- **File/path:** `supabase/migrations/20260629000009_allow_commissioner_pick_without_participant.sql`, `src/app/draft/DraftRoom.tsx`
- **What is wrong:** The current worktree contains a migration that lets the authoritative commissioner record a pick without a matching participant row. Until deployed, the live symptom remains possible. No database contract test covers this recovery case.
- **Why it matters:** Commissioner pick override can fail during precisely the identity drift/recovery situation it is meant to solve.
- **Recommended fix:** Add an RPC contract case for authoritative commissioner with no participant, deploy migration after local reset/contracts pass, and verify audit columns when `participant_id` is null.
- **Can safely fix now:** **Yes**, after Docker-backed database tests are available.

### I7. Chat has stale closure and recovery weaknesses

- **File/path:** `src/components/DraftChat.tsx`, `supabase/migrations/20260622040000_add_draft_chat.sql`
- **What is wrong:** The subscription effect captures `isOpen` and `onUnreadChange` but does not depend on them, so unread counts can increase while the panel is open. Initial-fetch/realtime races can duplicate messages. Fetch/send errors are largely silent. Commissioner announcements also require a participant row even when `commissioner_user_id` is authoritative.
- **Why it matters:** Chat state becomes misleading and shares the same commissioner-recovery failure mode as picks.
- **Recommended fix:** Use refs or correct effect dependencies, dedupe by message ID, surface errors, and authorize announcements by authoritative commissioner identity.
- **Can safely fix now:** **Yes, mostly**, with component and RPC tests.

### I8. Public/privileged endpoints have no abuse controls

- **File/path:** `src/app/api/auth/reset-password/route.ts`, `src/app/api/music/spotify-search/route.ts`, `src/app/api/music/youtube-search/route.ts`, `src/app/api/music/spotify-refresh/route.ts`, provider preview routes
- **What is wrong:** Password email generation and third-party search/refresh proxies have no IP/user throttling. Provider previews require a persistent user but are not rate-limited.
- **Why it matters:** These routes can consume Resend, Spotify, YouTube, Yahoo, ESPN, or other provider quotas and can be used for email abuse.
- **Recommended fix:** Add platform-level and per-user rate limits, response caching where safe, request-size limits, and observability. Keep account-enumeration-safe reset responses.
- **Can safely fix now:** **No.** A shared rate-limit backend/deployment choice is needed.

### I9. MFL API keys are sent in query strings

- **File/path:** `src/app/api/providers/mfl/preview/route.ts`, `src/lib/providerImport.ts`, `src/app/leagues/[slug]/seasons/new/NewSeasonForm.tsx`
- **What is wrong:** The browser sends `apiKey` as a query parameter to DraftHQ, and the server forwards it as a query parameter upstream.
- **Why it matters:** Query strings are more likely to appear in browser history, proxy logs, analytics, and monitoring traces.
- **Recommended fix:** POST credentials to DraftHQ in the body, redact logs, disable caching for credentialed previews, and follow MFL's safest supported upstream authentication mechanism.
- **Can safely fix now:** **Yes, bounded**, with API/client tests.

### I10. Member invitation creation is not transactional and account lookup is capped at 1,000 users

- **File/path:** `src/app/api/leagues/[leagueId]/members/route.ts`
- **What is wrong:** The route scans only the first 1,000 auth users, may create an auth user before invitation persistence succeeds, and then sends email as another separate side effect. Co-commissioner behavior also differs from the UI.
- **Why it matters:** Larger installations can mis-detect existing accounts; partial failures leave orphan auth users or invitations without email.
- **Recommended fix:** Use exact admin lookup where available or an indexed application profile lookup, introduce idempotency/status transitions, and move delivery to an outbox/worker.
- **Can safely fix now:** **No.** It needs a lifecycle/data model decision.

### I11. Spotify refresh tokens are stored in `localStorage`

- **File/path:** `src/lib/spotifyAuth.ts`, `src/app/api/music/spotify-callback/route.ts`, `src/app/spotify-popup-callback/page.tsx`
- **What is wrong:** OAuth access and refresh tokens are delivered to client JavaScript and persisted in `localStorage` without DraftHQ-user binding.
- **Why it matters:** Any XSS can exfiltrate long-lived Spotify credentials, and switching DraftHQ accounts in one browser shares the connection.
- **Recommended fix:** Keep refresh tokens server-side in an encrypted, user-bound store and issue a short-lived application session reference to the browser.
- **Can safely fix now:** **No.** This is an OAuth architecture change; music should remain lower priority than draft reliability.

## Polish and usability issues

### P1. Dashboard league rows contain nested links

- **File/path:** `src/app/dashboard/page.tsx`
- **What is wrong:** The entire league row is a `<Link>` and the "Team Owner" destination is another `<Link>` inside it.
- **Why it matters:** Nested anchors are invalid interactive markup and can produce inconsistent navigation, hydration, keyboard, and screen-reader behavior.
- **Recommended fix:** Use a non-anchor card with separate links, or make the main link an absolutely positioned sibling.
- **Can safely fix now:** **Yes.** Low risk.

### P2. Large client components concentrate unrelated state and effects

- **File/path:** `src/app/draft/DraftRoom.tsx` (~2,873 lines), `src/app/teams/TeamSetupForm.tsx` (~2,462 lines), `src/app/leagues/[slug]/teams/LeagueTeams.tsx`
- **What is wrong:** Draft state, presentation, audio, dialogs, commissioner controls, filtering, and responsive UI are coupled in a few components.
- **Why it matters:** Small changes can create stale closures, duplicate effects, and difficult regressions; this is already visible in lint and audio/chat issues.
- **Recommended fix:** After security/realtime work, extract domain hooks and presentational sections behind stable typed props. Do not redesign behavior during extraction.
- **Can safely fix now:** **No.** Structural refactor requiring characterization tests and user approval.

### P3. Several initial-load effects synchronously set state

- **File/path:** `src/app/draft/DraftRoom.tsx`, `src/app/join/JoinDraftForm.tsx`, `src/app/leagues/[slug]/seasons/new/NewSeasonForm.tsx`, `src/app/leagues/[slug]/teams/LeagueTeams.tsx`, `src/app/teams/TeamSetupForm.tsx`, `src/components/SongPicker.tsx`, `src/hooks/useLeagueWorkspace.ts`
- **What is wrong:** ESLint flags state updates directly in effects and one function accessed before declaration. These patterns are often symptoms of duplicated derived state or cascading renders.
- **Why it matters:** They can cause extra renders, flicker, and stale UI during navigation/recovery.
- **Recommended fix:** Initialize from props where possible, derive values during render, or isolate asynchronous synchronization with cancellation.
- **Can safely fix now:** **Yes individually**, but not as a blind lint cleanup.

### P4. Draft room render defines a component dynamically

- **File/path:** `src/app/draft/DraftRoom.tsx`
- **What is wrong:** `PickCard` is declared inside render; ESLint reports static-component violations.
- **Why it matters:** React treats the component identity as new on each render, which can reset child state and add work in a frequently updating room.
- **Recommended fix:** Move it to module scope with explicit props.
- **Can safely fix now:** **Yes.** Low risk with a render test.

### P5. Full visual smoke testing is not currently reproducible locally

- **File/path:** `playwright.config.ts`, `scripts/start-e2e-server.mjs`, local development environment
- **What is wrong:** E2E depends on local Supabase/Docker. Docker Desktop was unavailable in this audit. A stale Next dev process also held the dev lock without listening, and the isolated in-app browser could not reach the shell-hosted server.
- **Why it matters:** Responsive/mobile, redirects, loading states, and route interactions could not be independently confirmed in a clean browser session.
- **Recommended fix:** Make the E2E bootstrap detect/report stale Next locks, document Docker prerequisites, and run the full suite in CI on every release candidate.
- **Can safely fix now:** **Yes for diagnostics**, but the environment itself must be restored.

### P6. Base64 branding fallback can inflate league rows

- **File/path:** `src/app/leagues/[slug]/settings/LeagueSettingsForm.tsx`, `src/lib/leagueApi.ts`
- **What is wrong:** When storage upload is unavailable, branding can fall back to data URLs stored with league settings.
- **Why it matters:** Large images bloat database rows and every settings/workspace response, and bypass normal object lifecycle controls.
- **Recommended fix:** Make storage upload reliable and remove or tightly cap the data-URL fallback after migrating existing values.
- **Can safely fix now:** **No.** Existing data needs inspection/migration.

## Code quality and release-gate findings

### Q1. `npm test -- --run` fails despite the root unit tests passing

- **File/path:** `vitest.config.ts`, `.claude/worktrees/clever-kapitsa-2be807/tests/e2e/smoke.spec.mjs`
- **What is wrong:** Vitest discovers a nested worktree's Playwright smoke test. Root-scoped unit tests pass: **153/153**.
- **Why it matters:** The documented release command is not isolated from tooling worktrees and cannot be trusted as a clean gate.
- **Recommended fix:** Exclude `.claude/**` and other tooling worktrees from Vitest, ESLint, and TypeScript; keep E2E under Playwright only.
- **Can safely fix now:** **Yes.** Low risk configuration change.

### Q2. TypeScript validation fails in a test fixture

- **File/path:** `tests/unit/draftExport.test.ts`, `src/types/draft.ts`
- **What is wrong:** A `Pick` fixture is missing required `isLandmine`.
- **Why it matters:** `npx tsc --noEmit` is red, so the project lacks a clean whole-repo type gate even though Next build passes.
- **Recommended fix:** Update the fixture and add a dedicated `typecheck` script to the release command.
- **Can safely fix now:** **Yes.** Low risk.

### Q3. Root lint has 14 errors, while default lint is polluted by generated nested output

- **File/path:** `eslint.config.mjs` and files listed in P3/P4
- **What is wrong:** Default lint scans `.claude` worktrees/generated `.next` output. Excluding it reveals 14 real root errors, including effect-state, static-component, declaration-order, and unescaped-entity failures.
- **Why it matters:** The release gate is red and useful warnings are buried under thousands of irrelevant diagnostics.
- **Recommended fix:** Add tooling-worktree ignores, then resolve root errors deliberately and track warning reduction separately.
- **Can safely fix now:** **Yes for ignores and simple syntax issues**; effect refactors need targeted tests.

### Q4. Database, RLS, concurrency, integration, and E2E suites could not run

- **File/path:** `package.json`, `scripts/test-*-contracts.mjs`, `scripts/test-integration.mjs`, `tests/e2e/smoke.spec.mjs`
- **What is wrong:** `npm run test:db:contracts` fails at `supabase db reset` because the Docker Desktop Linux engine is unavailable. The dependent integration/E2E suites therefore cannot start.
- **Why it matters:** The highest-risk behaviors—RLS, commissioner authority, concurrent picks, undo/reset, realtime, and migrations—remain unverified against the current migration chain.
- **Recommended fix:** Restore Docker or run these suites in CI, then require all of `verify:release` before deployment.
- **Can safely fix now:** **Environment-dependent.** No code fix can substitute for running them.

### Q5. Build passes but is not a substitute for the failed gates

- **File/path:** `package.json`, Next.js configuration
- **What is wrong:** `npm run build` succeeds and compiles all routes, but the build does not catch the failing test fixture, RLS contracts, or lint errors.
- **Why it matters:** A green build can give a false production-readiness signal.
- **Recommended fix:** Keep build as one stage in a mandatory test/lint/typecheck/database/E2E pipeline.
- **Can safely fix now:** **Yes**, after the individual gates are repaired.

## Recommended improvements, in order

1. **Close authorization vulnerabilities:** remove stale participant commissioner authority, secure storage policies, revoke global bye-week writes, and protect ranking sync.
2. **Unify capabilities:** one authoritative draft commissioner source and one league capability model that explicitly includes/excludes co-commissioners per action.
3. **Make the release gate reproducible:** ignore nested tooling worktrees, fix the type fixture/lint errors, restore Docker, and run migrations/RLS/concurrency/integration/E2E in CI.
4. **Harden realtime recovery:** resubscribe with backoff, introduce a complete room revision, split compact draft state from the player catalog, and test disconnect/reconnect paths.
5. **Harden identity/provider flows:** user-bind Yahoo/Spotify tokens, add rate limits, stop putting MFL keys in URLs, and make invitation delivery idempotent.
6. **Then improve maintainability/polish:** split the largest components behind behavior tests, repair nested links/loading/error states, and optimize branding/media storage.

## Tests to add

### Database authorization and correctness

- Former commissioner participant cannot delete picks or reset after ownership transfer.
- Authoritative commissioner can pick, announce, undo, pause/resume, and reset with no participant row where intended.
- Ordinary participant and unrelated authenticated user cannot invoke any commissioner RPC.
- Co-commissioner capability matrix for settings, teams, members, seasons, integrations, draft controls, ownership transfer, and league deletion.
- Storage object tests for commissioner, assigned owner, unrelated member, unrelated league, and anonymous participant across insert/update/delete.

- Concurrent same-pick and adjacent-pick attempts, undo-versus-pick races, reset during an in-flight pick, and timer expiry races.

- Channel timeout/error resubscribes without focus/visibility events.
- Missed team/participant/invitation events are caught by the revision mechanism.
- Pick DELETE/reset refreshes only the affected draft.
- Multiple clients converge after pick, undo, edit, reset, pause/resume, extension, and reconnect.

- Yahoo tokens are inaccessible across DraftHQ users in one browser.

- Import RPCs enforce team/round bounds and commissioner scope independently of the client.

## Audit limitations

- Database findings are based on the complete migration chain, contract scripts, client/API callers, and a read-only check of current hosted state. The migrations could not be replayed locally because Docker was unavailable.
- The worktree already contained unrelated pending edits and a new migration before this audit. They were preserved and not modified.
- Visual/mobile behavior was reviewed from component structure and build output but needs a clean Playwright run for production sign-off.

## Proposed next decision

Before any large refactor, approve a narrow **security and release-gate stabilization phase**: C1-C4, storage policy tests, commissioner/co-commissioner capability decisions, configuration isolation, and restoration of Docker-backed contract tests. OAuth redesign and large component extraction should be separate follow-up phases.
